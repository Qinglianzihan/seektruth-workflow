状态：启动开发
核心思想：将《毛泽东选集》中的实事求是、矛盾分析、集中兵力、实践检验等方法论，转化为一套会调查研究、能因地制宜的 AI 辅助编程结构化纪律。

1. 项目宪章：不可动摇的核心
我们的项目区别于所有其他范式的根本点在于：它不是一个死板的规则包，而是一个活的、会“调查研究”的向导。 安装过程本身就是第一次“实事求是”的实践。

哲学原则	工程映射	强制程度
实事求是	安装时必须完成环境侦察，不盲目注入规则。	必须
矛盾分析	每项任务都必须识别并聚焦于唯一的核心矛盾。	必须
集中兵力	开发过程中，AI 的注意力必须被封锁在核心任务区域内。	必须
实践检验	所有代码产出都必须经过测试验证，失败即返回调查阶段。	必须
总结转化	每个任务周期都必须生成总结报告，沉淀为项目知识。	必须

3. 详细实施路线图（八阶段开发计划）
我们将分八个阶段完成开发，每个阶段的开发工作本身，都必须遵循“求是引擎”的工作流纪律。

Phase 0：初始化调研与本地化适配 (优先级最高)
目标：实现 stw init 命令，使其成为“会调查研究的安装程序”。

核心任务：

环境扫描器：实现项目类型（Node.js/Python），包管理器，测试框架的检测。

AI 工具探测器：检测用户使用的工具（Claude Code, Codex, Cline 等）及其配置文件路径。

能力扫描器 (MCP/Skill)：

解析各平台配置文件（.mcp.json，~/.claude.json等）。

扫描 skills/、commands/目录下的 Markdown 文件，解析出名字和描述。

（可选）尝试连接 MCP 服务器调用 tools/list，失败则标记为“待验证”。

报告生成器：生成结构化的《敌情侦察报告》，展示发现的能力、潜在冲突和建议。

本地化适配器：

根据报告决定启用/调整/禁用什么规则。

解决与其他技能（如 superpowers）的冲突。

生成定制化的 .stw/STW-Workspace.md 和 toae-config.json。

交付物：可运行的 stw init 命令，安装后项目根目录生成 .stw/ 文件夹及本地化配置。

Phase 1：基础框架与核心规范 (MVP-1)
目标：定义核心规范文件，搭建最简 CLI 壳，让 AI 能“读懂”我们的纪律。

核心任务：

规范文件编写：

STW-Workspace.md：核心工作流宣言，包含能力注册表区域。

Analysis-Template.md：阶段1“调查研究”报告模板。

Summary-Template.md：阶段5“总结报告”模板。

最简 CLI：实现 stw start 命令，它仅做两件事：

执行 stw detect（或加载已有的武器库清单）。

将 STW-Workspace.md 的内容和指令格式化输出，告诉用户“请把这些内容作为给 AI 的系统提示”。

交付物：发布 v0.1.0 到 npm，可通过 npx seektruth-workflow start 体验核心流程。

Phase 2：纪律引擎实现 (MVP-2)
目标：实现 stw next、stw status，强制流程流转。

核心任务：

状态机实现：用代码定义五阶段和流转规则。

交付物校验器：每个阶段跳转前，检查对应的文件（如调研报告）是否已生成。

专注封锁器：在进入阶段3时，stw next 命令读取任务中定义的“作战区域”，生成一份“封锁清单”注入到 AI 上下文。

交付物：完整的 stw 命令集（start， next， status， detect）。

Phase 3：知识沉淀系统
目标：让项目拥有记忆。

核心任务：

报告生成器：实现 stw report 命令，引导 AI 或用户填写《总结报告》。

记忆库：报告存储到 .stw/reports/，并按规则命名。

回忆加载：修改 stw start 的逻辑，在初始化上下文时，自动加载最近 N 份总结报告的摘要，作为“历史经验”注入。

交付物：项目开始积累知识，后续任务中 AI 能引用过去的经验。

Phase 4：武器库与环境深度集成
目标：让“能力侦探”更强大，能查询 MCP 工具详情，并深度适配 Claude Code。

核心任务：

MCP 深度集成：引入 @modelcontextprotocol/sdk，在扫描时尝试连接 MCP 服务器并调用 tools/list，获取精确的工具描述。

平台专用适配：为 Claude Code 编写最佳实践指南，利用其 hooks/commands 机制强化纪律（如在 PreToolUse hook 中检查文件路径是否在封锁清单内）。

交付物：“武器库”中 MCP 工具信息更详尽，Claude Code 用户体验更佳。

Phase 5：成本意识与Token经济
目标：引入“精兵简政”思想，追踪资源消耗。

核心任务：

日志解析器：从 Claude Code 等工具的日志中解析 token 消耗数据。

成本报告：在《总结报告》模板中加入“资源消耗”章节。

stw stats 命令：提供查看项目整体消耗趋势的功能。

交付物：用户和 AI 能直观看到每一次“战役”的资源成本。

Phase 6：多智能体协作（民主集中制）
目标：探索多 Agent 协作审查，防止 AI “一言堂”。

核心任务：

审查代理规范：编写一个独立的 审查员.md Agent 定义。

集成点设计：在 STW-Workspace.md 的阶段4，增加一条指令：“在此阶段，建议调用‘审查员’子代理对代码进行独立审查，并将审查结果作为实践检验的补充。”

提供示例：展示如何在 Claude Code 中配置和使用这个子代理。

交付物：项目支持可选的“民主集中制”代码审查流程。

Phase 7：生态与布道
目标：文档完善，社区建设。

核心任务：

文档站：使用 VitePress 搭建完整的用户和开发者文档。

示例项目：创建一个演示项目，展示用 STW 规范进行开发的全过程，并对比与不用工作流的差异。

发布推广：在相关社区（如 Claude 官方社区、Reddit、Twitter）分享。

交付物：项目拥有独立网站、完善的文档和至少一个示例项目。

4. 与 AI 协作开发的军规
自举 (Dogfooding)：从第一天起，我们就要用 STW 的纪律来开发 STW 自身。每次开发新功能，都新建一个会话，从 stw start 开始。

分拆战役：严格遵守“集中兵力”原则。一个会话只解决一个阶段内的一个子任务（如“只写分析报告模板”）。

强制测试：为核心CLI逻辑（状态机、校验器）编写自动化单元测试，并作为阶段4“实践检验”的硬性通过标准。

持续总结：每个开发会话结束，必须使用 stw report 完成阶段5，提交报告到仓库。这是项目的“战斗日志”。
