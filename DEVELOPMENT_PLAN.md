# “求是工作流” (SeekTruth Workflow, STW) 开发文档

> **文档版本**：1.0  
> **文档状态**：就绪，可执行  
> **读者**：与 AI 协作的人类开发者 + AI 编程代理  
> **使用方式**：将此文件作为项目最高规格说明。在每次开发会话中，先让 AI 阅读此文档，再根据当前阶段的任务开始编码。

---

## 第一部分：项目蓝图

### 1.1 项目简介

“求是工作流”是一个**开源、轻量、跨平台**的 AI 编程辅助工具。它将《毛泽东选集》中的哲学方法论，转化为一套 AI 编程代理必须遵守的结构化工作纪律，从而解决 AI 编程中常见的**上下文腐化、目标漂移、记忆缺失**等问题。

### 1.2 核心差异化

-   **哲学根基**：实事求是、矛盾分析、实践论。
-   **安装即调查**：安装过程中，工具会先侦察用户环境（已安装的 MCP、Skills 等），再生成本地化规则，不盲目注入。
-   **强纪律性**：强制执行五阶段工作流，禁止跳过。
-   **自我进化**：内建总结与知识沉淀机制，项目越用 AI 越“聪明”。

### 1.3 目标用户

-   使用 Claude Code、Codex CLI、Cline 等 AI 编程工具的开发者。
-   需要接手和维护复杂遗留系统（棕地项目）的团队。

## 第二部分：核心工作流与原则

### 2.1 三大原则

1.  **调查研究先行**：没有调查就没有发言权。AI 在编写任何代码前，必须先深入调研。
2.  **矛盾分析驱动**：任何任务都只有一个主要矛盾，必须抓住它并集中力量解决。
3.  **实践-认识-再实践**：真理的标准是实践。所有产出必须经过测试验证，失败则返回调查阶段。

### 2.2 五阶段强制工作流

这是 AI 在每次任务中必须遵守的**不可违背**的流程：

| 阶段 | 名称 | 核心要求 | 交付物 |
| :--- | :--- | :--- | :--- |
| 1 | **调查研究** | 阅读相关代码，分析项目背景，访谈用户。 | `敌情分析报告` |
| 2 | **抓住主要矛盾** | 与用户共同确定唯一核心任务，锁定目标。 | 任务聚焦声明(写入规范) |
| 3 | **集中优势兵力** | 只在核心任务相关区域修改，封锁其他文件。 | 专项代码变更 |
| 4 | **实践检验** | 强制测试。失败则**必须返回阶段1重新调查**。 | 测试通过报告 |
| 5 | **总结与转化** | 记录本次战役的认知迭代和解决方案，沉淀知识。 | `总结报告` |

## 第三部分：技术架构

### 3.1 总体形态

最终产物是一个可通过 `npx` 启动的 CLI 工具，安装在项目本地：

```bash
# 在现有项目中初始化求是工作流
npx stw init

# 在后续开发中，启动一次任务
stw start
```

### 3.2 项目文件结构

在用户项目中，安装完成后会增加以下文件：

```
用户项目根目录/
├── .stw/                         # 工作流控制目录（不侵入其他配置）
│   ├── STW-Workspace.md          # 核心规范 + 能力注册表
│   ├── Analysis-Template.md      # 调研报告模板
│   ├── Summary-Template.md       # 总结报告模板
│   ├── config.json               # 本地化适配配置（记录侦察结果和冲突决策）
│   └── reports/                  # 历史总结报告存档
└── .gitignore                    # 用户稍微修改，确保 .stw/reports 入库
```

### 3.3 技术选型

-   **运行时**：Node.js (≥18)，使用 ES Modules。
-   **依赖**：**零外部运行时依赖**。核心扫描逻辑只使用 Node.js 内置的 `fs`、`path` 等模块。仅在需要深度查询 MCP 工具时，可选引入 `@modelcontextprotocol/sdk`。
-   **包管理**：npm，最终包名 `seektruth-workflow`，命令为 `stw`。
-   **测试**：Node.js 内置 `node:test` + `assert`。

## 第四部分：开发路线图

我们将整个项目的开发分为 **8 个阶段 (Phase)**，每个阶段都要求我们**自己吃自己的狗粮**，即用 STW 的纪律来开发 STW。

### Phase 0：项目搭建与基础 CLI 壳

**目标**：创建仓库，完成基本 `package.json`，让 `stw` 命令可以跑起来。

**任务清单**：
-   [ ] 初始化 npm 包 `seektruth-workflow`。
-   [ ] 设置项目为 ES Module (`"type": "module"`)。
-   [ ] 创建 `bin/stw.js` 入口文件，使用 `#!/usr/bin/env node`。
-   [ ] 实现最小化 CLI 框架，能响应 `stw init` 和 `stw start` 命令，目前只输出一行字，证明命令能被识别。
-   [ ] 配置 `"bin"` 字段指向 `bin/stw.js`。
-   [ ] 编写 README 初稿。

**验收标准**：在项目目录执行 `npm link` 后，可以在任何终端输入 `stw init` 看到 “Init command not implemented yet” 字样。

### Phase 1：“能力侦探” —— 环境侦察核心

**目标**：实现 `stw init` 命令的侦察部分。能扫描用户 AI 工具和已安装能力，生成一份侦察报告打印在屏幕上。

**任务清单**：
-   [ ] 编写 `src/scout/project-detector.js`：检测当前项目技术栈（`package.json` 存在则判定为 Node.js 等）。
-   [ ] 编写 `src/scout/ai-tool-detector.js`：检测用户可能使用的 AI 工具（通过环境变量 `CLAUDE_CODE_VERSION`，或检查 `~/.claude` 目录等）。
-   [ ] 编写 `src/scout/mcp-scanner.js`：扫描常见的 MCP 配置文件（`.mcp.json`，`~/.claude.json`，`~/.cursor` 等），解析出已配置的 MCP 服务器名称。
-   [ ] 编写 `src/scout/skill-scanner.js`：扫描 `.claude/skills/`、`.agents/skills/` 等目录，读取 Markdown 文件头部的 YAML (`name`+`description`)。
-   [ ] 编写 `src/scout/report-generator.js`：将以上扫描结果汇总，生成格式化的终端输出。
-   [ ] 修改 `bin/stw.js`，将 `stw init` 连接至完整的侦察流程。

**验收标准**：在一个有 Claude Code 配置的电脑上，运行 `stw init` 能看到彩色的侦察报告，列出“项目类型：Node.js”，“已发现 MCP：xxx”，“已发现 Skills：xxx”等。

### Phase 2：“本地化适配” —— 生成初始文件

**目标**：扩展 `stw init`，使其根据 Phase 1 的侦察结果，在用户项目根目录生成 `.stw/` 文件夹和所有必要的规范文件及 `config.json`。

**任务清单**：
-   [ ] 设计 `config.json` 的数据结构（包含启用规则、冲突解决、武器库快照等）。
-   [ ] 编写 `src/adapters/rule-selector.js`：根据项目类型决定启用哪些规则变体。
-   [ ] 编写 `src/adapters/conflict-resolver.js`：检查发现与现有 Skill（如 superpowers）的重叠，生成警告和解决建议。
-   [ ] 编写 `src/adapters/file-writer.js`：根据模板和配置，生成 `.stw/` 下的所有文件。
-   [ ] 创建所有模板文件：
    -   `templates/STW-Workspace.md`
    -   `templates/Analysis-Template.md`
    -   `templates/Summary-Template.md`
-   [ ] 将 `stw init` 流程串起来：侦察 → 生成配置 → 写文件 → 打印成功信息。

**验收标准**：在一个空目录下运行 `stw init`，能够成功生成完整的 `.stw/` 目录，并且打开 `STW-Workspace.md` 可以看到本地化的信息（如“已发现你使用 Claude Code，以下 MCP 工具可用...”）。

### Phase 3：“阶段纪律引擎” —— 核心流程控制

**目标**：实现 `stw start`、`stw next`、`stw status` 命令，使其能强制执行五阶段工作流。

**任务清单**：
-   [ ] 设计并实现一个**状态机**模块 `src/engine/state-machine.js`，管理五个阶段的转换规则。
-   [ ] 实现 `stw status` 命令：读取当前进度（可从 `.stw/.progress` 文件或 `config.json` 中读取），显示当前阶段和已完成步骤。
-   [ ] 实现 `stw start` 命令：检查是否已 `init`；然后运行 `stw detect`（如果配置已过期）；最后输出指引，引导用户将 `STW-Workspace.md` 作为系统提示给 AI，并标记“阶段1”开始。
-   [ ] 实现 `stw next` 命令：
    -   检查当前阶段交付物是否存在。
    -   如果存在，更新进度到下一阶段，并输出下一阶段的指引。
    -   如果交付物缺失，输出错误，告诉用户需完成什么。
-   [ ] 编写单元测试，覆盖状态机的流转逻辑和交付物检查逻辑。

**验收标准**：可以在一个已安装的项目中，通过 `stw start` 开始，然后创建一个空的 `Analysis-Template.md`，运行 `stw next` 能成功进入阶段2，并锁定阶段3需要先定义聚焦区域。

### Phase 4：“专注封锁”与“知识沉淀”

**目标**：实现阶段3的“封锁清单”生成，以及阶段5的总结报告自动管理与回忆加载。

**任务清单**：
-   [ ] 在 `STW-Workspace.md` 中定义“作战区域”声明格式（如 `<!-- ATTACK_ZONE: src/orders/* -->`）。
-   [ ] 修改 `stw next`：当进入阶段3时，自动从规范中解析作战区域，生成 `.stw/lockdown.json`，并输出“已封锁其他区域”的强烈提示。
-   [ ] 实现 `stw report` 命令：引导用户（或由 AI 调用）填写 `Summary-Template.md` 并保存到 `.stw/reports/` 下。
-   [ ] 修改 `stw start`：在初始化 AI 上下文时，自动读取最近 3 份总结报告的摘要，并添加到给 AI 的指令中。

**验收标准**：可以完整走完一个五阶段任务流程，阶段3时 AI 明确知道不可修改 `lockdown.json` 外的文件，任务结束后能用 `stw report` 生成报告。

### Phase 5：深度 MCP 集成与 Token 追踪

**目标**：利用 MCP SDK 获取精确的工具描述；引入“精兵简政”的 token 消耗意识。

**任务清单**：
-   [ ] 可选引入 `@modelcontextprotocol/sdk`，在侦察阶段尝试连接 MCP 服务器并调用 `tools/list` 获取详细能力。
-   [ ] 修改 `Summary-Template.md`，加入“资源消耗估算”章节。
-   [ ] 实现 `stw stats` 命令，解析日志或手动输入，统计项目整体 token 消耗趋势。
-   [ ] 如果平台提供 token 用量 API，尝试从 Claude Code 等工具的会话日志中提取数据。

**验收标准**：侦察报告中 MCP 工具不再是空名字，而是带有描述；总结报告里可以看到 AI 自查的 token 消耗量。

### Phase 6：多智能体协作（民主集中制）支持

**目标**：在规范中嵌入多 Agent 审查的指导，并为 Claude Code 等平台提供样本。

**任务清单**：
-   [ ] 编写一个独立的“审查员”子代理 Markdown 文件（如 `审查员.md`），定义其审查职责。
-   [ ] 在 `STW-Workspace.md` 的阶段4，明确建议：“调用审查员子代理对代码进行独立审查”。
-   [ ] 为 Claude Code 编写 `审查员.claude.md` 样本，展示如何在项目中配置和使用。

**验收标准**：在 Claude Code 项目中，用户可以按指引配置审查员子代理，并在阶段4看到明确的审查提醒。

### Phase 7：文档、发布与生态

**目标**：完善一切，准备向世界发布。

## 第五部分：与 AI 协作开发的纪律

### 5.1 我们要“吃自己的狗粮”

从 Phase 0 结束后，每个新功能的开发，都必须在我们自己的 `.stw/` 管理下进行。也就是说：
1.  在开始一个新 Phase 的功能前，先运行 `stw start`，让 AI 研究我们的代码库。
2.  用 `stw next` 推进阶段，强制自己完成调查、聚焦、测试和总结。

### 5.2 拆分小战役

严格遵守“集中优势兵力”原则。**一个会话只解决一个阶段内的一个独立模块。** 例如：“这次会话我们只完成 `project-detector.js`，不接受其他需求。”

### 5.3 强制测试

所有的 CLI 逻辑模块（状态机、文件扫描、配置生成）都必须有对应的单元测试。测试是阶段4“实践检验”的硬性门槛，**不通过不准进入下一阶段**。

### 5.4 持续总结

每一个开发会话结束时，都必须运行 `stw report`（或手动编写总结），将这次开发的认知迭代记录下来，提交到 Git。这是项目的“记忆”。

## 第六部分：第一个开发会话指南

假设你现在正坐在电脑前，旁边是 AI 编程工具（比如 Claude Code）。请按以下步骤开始：

1.  **创建项目目录**：
    ```bash
    mkdir seektruth-workflow
    cd seektruth-workflow
    git init
    ```

2.  **将此文档保存到项目根目录**：
    另存为 `DEVELOPMENT_PLAN.md`。

3.  **开启 AI 会话，并输入**：
    > “请阅读项目根目录下的 `DEVELOPMENT_PLAN.md`。这是我们项目的开发规格说明书。我们从 **Phase 0** 开始执行。请先帮我初始化一个 Node.js 项目，并完成 Phase 0 的任务清单。一步一步来，每完成一步请向我确认。”

4.  **开始协作**：AI 会引导你创建 `package.json`，编写最基础的 CLI 入口文件。你负责执行命令和测试。

**记住，你是决策者，AI 是你忠实的工程兵。** 用“求是工作流”的哲学来管理你的 AI 工程兵，一起把这个项目从蓝图变成现实。
